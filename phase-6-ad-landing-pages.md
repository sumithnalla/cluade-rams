## Two flags to resolve before this runs at scale

**1. Reviews carousel location mismatch.** In the sample page, several testimonials are shown with city localities that don't match their origin in `cities.js` (e.g. Anand M. is the Bangalore/Whitefield testimonial in `cities.js`, but appears as "Madhapur" — a Hyderabad locality — on the Hyderabad landing page, under the Google logo with 5 stars). This reads as a real, geographically-specific Google review when it isn't one. Pick one: (a) hold the carousel back until Google Business Profiles are live and real reviews can be pulled in, or (b) keep using the existing testimonial copy but drop the Google-branded styling and keep each quote's real city/area attached. Don't relabel real customers' locations to fill out cards.

**2. Tailwind via `cdn.tailwindcss.com`.** Fine for the one-off prototype, not for 30 production ad-landing pages — it ships the full JIT compiler to the browser and recompiles on load, which works against page speed and Quality Score (the same concern Phase 5 already flagged). Recommend compiling Tailwind to a static CSS file as part of the build step, or porting the classes used here onto the existing `main.css` token system, before deploying.

## Indexing
Confirmed: **noindex**. Add `<meta name="robots" content="noindex, nofollow">` to every one of the 30 pages. Do not add `/lp/` URLs to `buildSitemap()`. No `robots.txt` change needed — a disallow rule would stop Google from ever seeing the noindex tag, so leave crawling open and rely on the meta tag alone.

## Lead capture — currently broken, needs to be added

The sample page's `setupLeadForms()` only does `e.preventDefault()` and opens a WhatsApp deep link — it never calls the Apps Script. As written, no lead would ever reach the sheet or trigger the routed email. Add this on top of (not instead of) the existing WhatsApp redirect, since both serve a purpose — the sheet/email is your record and routing, WhatsApp is the instant-response channel for the lead:

- **Hidden field, baked at build time per page** (not user input): `Source`, value = `` `${city.name} | ${adGroup.name} Landing Page` `` — e.g. `"Hyderabad | LED Screen & Video Wall Landing Page"`. The Apps Script does a lowercase substring match on city name, so as long as the city name is in there, routing to the correct city manager email works; the ad-group name riding along gives Sumi visibility into which campaign converted.
- **POST target:** `https://script.google.com/macros/s/AKfycbwQBnK4brEDrtnHe_-HNZjdJABPy_4XIQTu1WATjrwqwVc2kcR-i6s8XVnECMLr7rktOw/exec`
- **Body params, exact case:** `Name`, `Number`, `Source` (matches `e.parameter.Name` etc. in the Apps Script — anything else will silently fall through to the `"N/A"` / `"No source provided"` fallbacks).
- **CORS workaround:** Apps Script web apps don't return CORS headers for `fetch()` by default. Either (a) `fetch(url, { method: 'POST', mode: 'no-cors', body: formData })` — fire-and-forget, can't read the JSON response back, or (b) the classic hidden-iframe form-submit trick. Either is fine for lead capture where you don't need a client-side success/fail state.
- Keep the existing WhatsApp-redirect behavior after the submission fires.

Both the inline hero form and the modal form on each page need this wired — they're two separate `<form data-lead-form>` instances in the sample.
